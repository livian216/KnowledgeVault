# 将本地已有工程同步到 GitHub 仓库的操作备忘

## 1. 适用场景

本文档用于记录一种常见情况：

> 本地已经创建了一个工程目录，并且已经开发了一段时间，目录中已经有若干代码、配置文件、文档或素材文件；同时，GitHub 上也已经新建了一个对应的空仓库。现在希望把本地工程完整同步到 GitHub 云端仓库中。

例如：

- 本地工程目录：`FIREUAV-DEMO`
- GitHub 仓库名称：`FIREUAV-DEMO`
- 本地目录中已经有代码文件
- GitHub 仓库刚刚创建，基本为空

这种情况下，不需要重新创建工程，也不需要把代码手动复制到 GitHub 页面中，只需要在本地工程目录内使用 Git 命令完成初始化、提交和推送即可。

---

## 2. 操作前需要准备什么

在开始操作前，需要确认以下几件事。

### 2.1 本地已经安装 Git

在终端中执行：

```bash
git --version
```

如果能看到类似下面的输出，说明 Git 已经安装：

```bash
git version 2.xx.x
```

如果提示找不到 `git` 命令，需要先安装 Git。

### 2.2 GitHub 上已经创建好空仓库

进入 GitHub，创建一个新的仓库，例如：

```text
FIREUAV-DEMO
```

建议注意：

1. 仓库名称最好和本地工程目录名称保持一致；
2. 如果是第一次同步本地已有代码，GitHub 仓库可以保持为空；
3. 可以不勾选自动创建 `README.md`、`.gitignore`、`LICENSE`，这样后续推送会更简单。

如果已经勾选了这些文件，也可以处理，后文会说明。

### 2.3 确认本地工程目录路径

例如本地工程目录为：

```text
D:\Workspaces\FIREUAV-DEMO
```

或者：

```text
/home/your-username/workspaces/FIREUAV-DEMO
```

后续所有 Git 命令都需要在这个工程目录内部执行。

---

## 3. 标准流程：本地已有工程，远端为空仓库

这是最推荐、最常见的操作流程。

---

### 第一步：进入本地工程目录

Windows CMD 或 PowerShell 示例：

```bash
cd /d D:\Workspaces\FIREUAV-DEMO
```

Git Bash 示例：

```bash
cd /d/Workspaces/FIREUAV-DEMO
```

Linux 示例：

```bash
cd /home/your-username/workspaces/FIREUAV-DEMO
```

进入目录后，可以查看当前目录下的文件：

```bash
ls
```

Windows CMD 中也可以使用：

```bash
dir
```

---

### 第二步：初始化 Git 仓库

在本地工程目录内执行：

```bash
git init
```

执行后，当前目录会变成一个 Git 仓库。

Git 会在项目目录下创建一个隐藏目录：

```text
.git/
```

这个目录用于记录 Git 版本管理信息，不要手动删除或修改。

---

### 第三步：设置主分支名称为 main

建议统一使用 `main` 作为主分支名称：

```bash
git branch -M main
```

这样本地分支名称会变成 `main`，与 GitHub 当前默认分支习惯保持一致。

---

### 第四步：创建 `.gitignore` 文件

在正式提交之前，建议先创建 `.gitignore` 文件，用于排除不应该上传到 GitHub 的文件。

例如：

```bash
notepad .gitignore
```

Linux 或 Git Bash 下可以使用：

```bash
touch .gitignore
```

然后编辑 `.gitignore`，写入以下内容作为基础版本：

```gitignore
# Python cache
__pycache__/
*.pyc
*.pyo
*.pyd

# Python virtual environments
.venv/
venv/
env/

# Logs
*.log
logs/

# Node.js / frontend
node_modules/
dist/
build/
.vite/

# IDE settings
.vscode/
.idea/

# OS files
.DS_Store
Thumbs.db

# Environment variables and secrets
.env
.env.*
*.key
*.pem

# Large model files or generated outputs
*.pt
*.pth
*.onnx
*.rknn
*.engine

# Datasets and temporary outputs
data/
datasets/
outputs/
temp/
tmp/
```

`.gitignore` 的作用是告诉 Git：

> 哪些文件不需要纳入版本管理，也不要上传到 GitHub。

通常不建议上传：

- 虚拟环境；
- 缓存文件；
- 日志文件；
- 临时文件；
- 编译产物；
- 大型数据集；
- 模型权重文件；
- 密钥、账号、密码、Token 等敏感信息。

---

### 第五步：查看当前 Git 状态

执行：

```bash
git status
```

这个命令用于查看当前有哪些文件还没有被 Git 管理。

第一次执行时，通常会看到很多 `Untracked files`，表示这些文件还没有被添加到 Git 暂存区。

---

### 第六步：添加所有需要提交的文件

执行：

```bash
git add .
```

含义是：

> 把当前目录下所有未被 `.gitignore` 排除的文件添加到 Git 暂存区。

如果只想添加某个文件，也可以使用：

```bash
git add 文件名
```

例如：

```bash
git add README.md
```

---

### 第七步：提交本地代码

执行：

```bash
git commit -m "Initial commit"
```

也可以写得更明确一些：

```bash
git commit -m "Initial commit for FIREUAV-DEMO"
```

这里的 `commit` 可以理解为一次本地版本快照。

注意：

`git commit` 只是提交到本地 Git 仓库，还没有上传到 GitHub。

---

### 第八步：绑定 GitHub 远程仓库

进入 GitHub 对应仓库页面，复制仓库地址。

GitHub 仓库地址一般有两种形式。

#### 方式一：HTTPS 地址

格式类似：

```text
https://github.com/你的用户名/FIREUAV-DEMO.git
```

绑定命令：

```bash
git remote add origin https://github.com/你的用户名/FIREUAV-DEMO.git
```

例如：

```bash
git remote add origin https://github.com/你的用户名/FIREUAV-DEMO.git
```

#### 方式二：SSH 地址

如果本地已经配置过 GitHub SSH Key，可以使用 SSH 地址。

格式类似：

```text
git@github.com:你的用户名/FIREUAV-DEMO.git
```

绑定命令：

```bash
git remote add origin git@github.com:你的用户名/FIREUAV-DEMO.git
```

例如：

```bash
git remote add origin git@github.com:你的用户名/FIREUAV-DEMO.git
```

---

### 第九步：检查远程仓库是否绑定成功

执行：

```bash
git remote -v
```

如果看到类似下面的内容，说明绑定成功：

```bash
origin  https://github.com/你的用户名/FIREUAV-DEMO.git (fetch)
origin  https://github.com/你的用户名/FIREUAV-DEMO.git (push)
```

或者：

```bash
origin  git@github.com:你的用户名/FIREUAV-DEMO.git (fetch)
origin  git@github.com:你的用户名/FIREUAV-DEMO.git (push)
```

---

### 第十步：推送本地代码到 GitHub

如果 GitHub 仓库是空仓库，执行：

```bash
git push -u origin main
```

这条命令的含义是：

> 将本地 `main` 分支推送到远程 `origin` 仓库，并建立本地分支和远程分支的默认关联关系。

执行成功后，刷新 GitHub 仓库页面，就可以看到本地项目文件已经上传到 GitHub。

---

## 4. 最简命令汇总

如果 GitHub 仓库是空的，可以直接按下面顺序执行。

```bash
cd /d D:\Workspaces\FIREUAV-DEMO

git init
git branch -M main

git status

git add .
git commit -m "Initial commit for FIREUAV-DEMO"

git remote add origin https://github.com/你的用户名/FIREUAV-DEMO.git
git remote -v

git push -u origin main
```

需要把下面这一行替换成自己的真实 GitHub 仓库地址：

```bash
https://github.com/你的用户名/FIREUAV-DEMO.git
```

---

## 5. 后续日常开发如何继续同步

第一次推送成功后，后续每次修改代码，一般只需要执行以下命令。

### 5.1 查看修改状态

```bash
git status
```

### 5.2 添加修改文件

```bash
git add .
```

### 5.3 提交本地修改

```bash
git commit -m "说明本次修改内容"
```

例如：

```bash
git commit -m "Add fire scene demo overlay module"
```

### 5.4 推送到 GitHub

```bash
git push
```

第一次已经执行过：

```bash
git push -u origin main
```

所以后续通常只需要：

```bash
git push
```

---

## 6. 如果 GitHub 远程仓库不是空的怎么办

如果创建 GitHub 仓库时勾选了：

- `README.md`
- `.gitignore`
- `LICENSE`

那么 GitHub 远程仓库就不是完全空的。

这种情况下，直接执行：

```bash
git push -u origin main
```

可能会报错。

常见报错类似：

```text
rejected
Updates were rejected because the remote contains work that you do not have locally.
```

意思是：

> 远程仓库里已经有本地没有的内容，需要先把远程内容拉取下来，再推送。

可以执行：

```bash
git pull origin main --rebase
```

然后再推送：

```bash
git push -u origin main
```

如果拉取过程中出现冲突，需要先解决冲突文件，然后执行：

```bash
git add .
git rebase --continue
```

最后再推送：

```bash
git push -u origin main
```

---

## 7. 如果已经添加过错误的远程仓库地址怎么办

可以先查看远程地址：

```bash
git remote -v
```

如果地址写错了，可以修改：

```bash
git remote set-url origin 新的仓库地址
```

例如：

```bash
git remote set-url origin https://github.com/你的用户名/FIREUAV-DEMO.git
```

再次检查：

```bash
git remote -v
```

---

## 8. 如果提示 origin 已经存在怎么办

执行：

```bash
git remote add origin https://github.com/你的用户名/FIREUAV-DEMO.git
```

时，如果提示：

```text
remote origin already exists
```

说明当前仓库已经绑定过一个名为 `origin` 的远程仓库。

可以选择修改远程地址：

```bash
git remote set-url origin https://github.com/你的用户名/FIREUAV-DEMO.git
```

或者先删除再重新添加：

```bash
git remote remove origin
git remote add origin https://github.com/你的用户名/FIREUAV-DEMO.git
```

---

## 9. 如果提示没有配置用户名和邮箱怎么办

第一次使用 Git 提交时，可能会提示：

```text
Please tell me who you are.
```

需要配置 Git 用户名和邮箱：

```bash
git config --global user.name "你的名字"
git config --global user.email "你的邮箱"
```

例如：

```bash
git config --global user.name "你的名字"
git config --global user.email "xxx@example.com"
```

然后重新提交：

```bash
git commit -m "Initial commit"
```

查看当前配置：

```bash
git config --global --list
```

---

## 10. 如果文件太大推送失败怎么办

GitHub 对单个文件大小有限制。

如果项目中包含以下文件，可能导致推送失败：

- 大模型权重文件；
- 视频文件；
- 数据集压缩包；
- 大型图片素材；
- 编译生成的大文件；
- 临时输出结果。

建议不要直接把这些文件提交到 GitHub。

可以把它们加入 `.gitignore`，例如：

```gitignore
*.mp4
*.avi
*.mov
*.zip
*.rar
*.7z
*.pt
*.pth
*.onnx
*.rknn
data/
datasets/
outputs/
```

如果大文件已经被 `git add` 了，但还没有 commit，可以取消暂存：

```bash
git restore --staged 大文件名
```

如果大文件已经 commit 了，处理会更麻烦，需要从 Git 历史中移除。新人阶段建议在第一次提交前先认真检查 `.gitignore`。

## 11. 推荐的提交信息写法

提交信息不要总是写：

```bash
git commit -m "update"
```

建议简洁说明本次修改内容，例如：

```bash
git commit -m "Add project initialization files"
git commit -m "Add fire scene base map generation script"
git commit -m "Fix video export path issue"
git commit -m "Update README with setup instructions"
git commit -m "Refactor overlay rendering module"
```

好的提交信息可以帮助以后快速回顾项目历史。

## 12. 新人需要理解的几个概念

### 12.1 本地工程目录

就是电脑上实际存放代码的文件夹，例如：

```text
FIREUAV-DEMO/
```

### 12.2 本地 Git 仓库

执行 `git init` 后，本地目录就变成了 Git 仓库。

Git 会通过 `.git/` 隐藏目录记录版本历史。

### 12.3 GitHub 远程仓库

GitHub 上创建的云端仓库，用于远程备份和协作。

### 12.4 git add

把文件加入暂存区，表示这些文件准备进入下一次提交。

### 12.5 git commit

在本地生成一次版本记录。

### 12.6 git push

把本地提交推送到 GitHub 云端仓库。

### 12.7 origin

`origin` 是远程仓库的默认别名。

例如：

```bash
git remote add origin https://github.com/你的用户名/FIREUAV-DEMO.git
```

意思是把这个 GitHub 仓库命名为 `origin`，以后推送时可以直接使用：

```bash
git push origin main
```

## 13. 建议的完整工作习惯

对于一个已经开发了一定阶段的本地项目，建议按以下习惯管理。

### 13.1 第一次同步

```bash
git init
git branch -M main
```

创建 `.gitignore`，排除不需要上传的文件。

```bash
git add .
git commit -m "Initial commit"
git remote add origin 仓库地址
git push -u origin main
```

### 13.2 后续每次开发完成一个小功能

```bash
git status
git add .
git commit -m "Add xxx feature"
git push
```

### 13.3 每次准备修改前先看状态

```bash
git status
```

### 13.4 每次推送前确认没有误提交敏感文件

重点检查：

- `.env`
- 密码文件
- 私钥文件
- 大模型文件
- 数据集文件
- 临时日志
- 本地配置文件

## 14. 一份推荐的 README.md 基础模板

第一次同步项目时，建议在项目根目录下创建 `README.md`，用于说明项目用途。

示例：

```markdown

## 项目简介

本项目用于消防全景融合与态势感知无人机 Demo 的快速原型开发。

## 主要功能

- 生成消防救援场景演示素材
- 叠加火点、烟雾、风险区域等可视化元素
- 输出用于汇报展示的图片或视频素材

## 环境要求

- Python 3.x
- OpenCV
- NumPy
- 其他依赖见 requirements.txt

```bash
pip install -r requirements.txt

本项目为 Demo 原型工程，主要用于汇报展示和方案验证。
```

## 15. 推荐的新手操作顺序

如果完全没有 Git 经验，可以按照下面顺序做：

1. 进入本地工程目录；
2. 执行 `git init`；
3. 创建 `.gitignore`；
4. 执行 `git status` 检查文件；
5. 执行 `git add .`；
6. 执行 `git commit -m "Initial commit"`；
7. 执行 `git remote add origin 仓库地址`；
8. 执行 `git push -u origin main`；
9. 打开 GitHub 仓库页面检查文件是否上传成功；
10. 后续修改后，重复 `git add .`、`git commit`、`git push`。

## 17. 最终总结

当一个本地工程已经开发了一段时间，需要同步到 GitHub 时，核心流程可以概括为：

```bash
cd 项目目录
git init
git branch -M main
git add .
git commit -m "Initial commit"
git remote add origin GitHub仓库地址
git push -u origin main
```

实际操作时，最需要注意的是：

1. 先写好 `.gitignore`，避免上传无关文件、大文件和敏感文件；
2. 确认 GitHub 仓库地址正确；
3. 如果远程仓库不是空的，需要先 `git pull origin main --rebase`；
4. 后续日常同步只需要 `git add .`、`git commit`、`git push`。

掌握以上流程后，就可以比较规范地把本地已有项目同步到 GitHub，并在后续开发中持续维护代码版本。
