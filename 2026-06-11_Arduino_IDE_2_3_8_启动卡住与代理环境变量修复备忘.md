# Arduino IDE 2.3.8 启动卡在启动页面的问题处理备忘

## 1. 问题现象

在 Windows 11 环境下，Arduino IDE 2.3.8 打开后长时间停留在启动页面，无法快速进入正常的开发界面。

此前为了修复 Codex 客户端登录问题，曾经在 PowerShell 中设置过系统代理环境变量，例如：

```powershell
setx HTTP_PROXY "http://127.0.0.1:7890"
setx HTTPS_PROXY "http://127.0.0.1:7890"
```

其中 `7890` 是当时本机 VPN/代理软件使用的本地代理端口。不同代理软件的端口可能不同，需要根据自己的实际情况调整。

## 2. 问题原因判断

Arduino IDE 2.x 启动时，需要和本机内部组件进行通信，其中会涉及 `localhost`、`127.0.0.1` 等本机回环地址。

如果 Windows 用户环境变量中设置了 `HTTP_PROXY`、`HTTPS_PROXY` 等代理变量，但没有明确设置本机地址不走代理，那么 Arduino IDE 可能会把本机内部通信也错误地交给代理处理，导致启动过程异常缓慢，甚至长时间停留在启动页面。

本次排查中，通过查看环境变量，确认电脑中确实存在之前为 Codex 登录问题设置的代理环境变量，因此该问题与代理配置高度相关。

## 3. 解决思路

不要直接删除 Arduino IDE 的缓存目录，也不要直接删除 Arduino 的板卡包目录。

尤其不建议直接删除：

```powershell
C:\Users\<你的用户名>\AppData\Local\Arduino15
```

因为该目录通常保存 Arduino IDE 已安装的开发板支持包、工具链、索引文件等内容。如果之前已经配置过 Teensy 开发环境，直接删除该目录可能导致 Teensy 相关依赖丢失，需要重新配置。

本次采用的安全处理思路是：

1. 关闭残留的 Arduino IDE / arduino-cli 进程；
2. 检查当前是否存在代理环境变量；
3. 保留原来的 `HTTP_PROXY` / `HTTPS_PROXY`，避免影响 Codex；
4. 额外增加 `localhost` 代理绕过规则，让 Arduino IDE 的本机内部通信不再走代理。

## 4. 具体执行步骤

### 第一步：关闭残留 Arduino 进程

在 PowerShell 中执行：

```powershell
Get-Process | Where-Object {
    $_.ProcessName -like "*Arduino*" -or $_.ProcessName -like "*arduino-cli*"
} | Stop-Process -Force
```

该步骤用于确保之前卡住的 Arduino IDE 或 arduino-cli 进程已经完全退出，避免旧进程继续占用资源或保留异常状态。

### 第二步：查看是否存在代理环境变量

执行：

```powershell
Get-ChildItem Env: | Where-Object {
    $_.Name -match "proxy"
} | Format-Table -Auto
```

如果能看到类似下面的内容，说明系统当前存在代理环境变量：

```powershell
HTTP_PROXY   http://127.0.0.1:7890
HTTPS_PROXY  http://127.0.0.1:7890
```

这与之前为解决 Codex 登录问题设置代理的操作相吻合。

### 第三步：增加 localhost 代理绕过规则

执行：

```powershell
setx no_proxy "localhost,127.0.0.1,::1"
setx NO_PROXY "localhost,127.0.0.1,::1"
```

这两条命令用于告诉相关程序：访问 `localhost`、`127.0.0.1`、`::1` 这些本机回环地址时，不要走代理。

其中：

- `localhost`：本机主机名；
- `127.0.0.1`：IPv4 本机回环地址；
- `::1`：IPv6 本机回环地址。

### 第四步：关闭 PowerShell 并重新打开 Arduino IDE

执行 `setx` 后，需要关闭当前 PowerShell、Arduino IDE 等已经打开的窗口，然后重新打开 Arduino IDE。

原因是 `setx` 写入的是新的用户环境变量配置，它通常只会对之后新启动的进程生效。已经打开的 PowerShell 或已经运行的程序，不一定能立即读取到新设置。

## 5. 实际解决效果

按照上述步骤处理后，重新打开 Arduino IDE 2.3.8，启动速度恢复正常，可以快速进入开发界面。

这说明本次 Arduino IDE 启动慢的问题，大概率是由于代理环境变量影响了 Arduino IDE 的本机内部通信，而通过 `no_proxy` / `NO_PROXY` 排除本机回环地址后，问题得到解决。

## 6. 该方法是否长期有效

本次解决方式是长期有效的，不是只对当前 PowerShell 会话临时有效。

原因是使用的是：

```powershell
setx no_proxy "localhost,127.0.0.1,::1"
setx NO_PROXY "localhost,127.0.0.1,::1"
```

`setx` 会把环境变量写入 Windows 的用户环境变量配置中。写入成功后，对之后新启动的程序长期生效，包括电脑重启之后也会继续保留。

需要注意的是：

1. `setx` 对当前已经打开的 PowerShell 窗口不一定立即生效；
2. 关闭并重新打开 PowerShell 或相关软件后，新环境变量才会被新进程读取；
3. 电脑重启后不会失效；
4. 只有当用户手动删除该环境变量、修改系统环境变量，或者某些软件覆盖了该配置时，才可能失效。

因此，本次操作属于用户级环境变量的长期配置修复，而不是一次性的临时修复。

## 7. 后续验证命令

如果以后想确认配置是否还存在，可以重新打开一个新的 PowerShell，然后执行：

```powershell
Get-ChildItem Env: | Where-Object {
    $_.Name -match "proxy"
} | Format-Table -Auto
```

正常情况下，应能看到类似：

```powershell
HTTP_PROXY   http://127.0.0.1:7890
HTTPS_PROXY  http://127.0.0.1:7890
NO_PROXY     localhost,127.0.0.1,::1
```

如果能看到 `NO_PROXY` 或 `no_proxy` 中包含：

```text
localhost,127.0.0.1,::1
```

说明本机代理绕过规则仍然存在。

## 8. 结论

Arduino IDE 2.3.8 启动卡在启动页面，并不一定是 Arduino IDE 本身损坏，也不一定需要删除 Arduino 的缓存目录或重新安装 Teensy 开发环境。

在本次场景中，问题是由于之前为 Codex 设置了代理环境变量，但没有设置本机回环地址的代理绕过规则，导致 Arduino IDE 启动时的本机内部通信受到代理影响。

最终通过增加：

```powershell
setx no_proxy "localhost,127.0.0.1,::1"
setx NO_PROXY "localhost,127.0.0.1,::1"
```

成功解决了 Arduino IDE 启动慢的问题，同时没有破坏已有的 Arduino / Teensy 开发环境。

该修复方式对新启动的程序长期有效，电脑重启后仍然有效。
