# KnowledgeVault

## 1. 项目简介

`KnowledgeVault` 是一个用于集中管理个人开发、学习与工作备忘笔记的 Markdown 知识库工程。

本工程当前阶段的主要目标是：将日常开发调试、技术学习、环境配置、问题排查、项目经验、工具使用方法等内容，以 Markdown 文档的形式统一保存到本地目录，并通过 Git/GitHub 实现云端备份与版本管理。

现阶段，`KnowledgeVault` 不依赖复杂的知识库系统，也不绑定特定笔记软件。所有笔记均以普通 Markdown 文件保存，便于长期维护、跨平台查看、版本追踪和后续扩展。

## 2. 建立本工程的目的

在日常开发和学习过程中，经常会产生大量零散信息，例如：

- Linux、Docker、Git、Python、C++ 等环境配置记录；
- 开发板、嵌入式设备、边缘计算平台的调试过程；
- 模型训练、模型部署、工程集成过程中的问题排查；
- 软件工具、命令行操作、第三方库使用方法；
- 项目方案、技术路线、开发计划和阶段总结；
- 个人学习过程中形成的知识点、理解和备忘。

这些内容如果只散落在聊天记录、临时文件、截图或不同目录中，后续很难快速查找和复用。因此，本工程用于将这些内容沉淀为可持续维护的个人知识资产。

## 3. 当前管理方式

当前阶段采用较轻量的管理方式：

1. 所有笔记以 Markdown 文件形式保存；
2. 本地目录作为主要编辑和维护位置；
3. Git 用于记录变更历史；
4. GitHub 用于云端备份和跨设备同步；
5. 暂不引入复杂的数据库、前端系统或大模型检索功能。

这种方式简单、稳定、迁移成本低，也适合作为后续智能知识库系统的基础数据层。

## 4. 推荐目录结构

当前可以先采用如下目录结构：

```text
KnowledgeVault/
├── README.md
├── notes/
│   ├── development/
│   ├── learning/
│   ├── tools/
│   ├── troubleshooting/
│   └── projects/
├── templates/
└── assets/
```

各目录说明如下：

| 目录 | 说明 |
|---|---|
| `notes/` | 存放主要 Markdown 笔记 |
| `notes/development/` | 开发过程、工程实践、代码相关记录 |
| `notes/learning/` | 系统学习笔记、概念理解、知识总结 |
| `notes/tools/` | Git、Docker、VS Code、Codex、开发工具等使用记录 |
| `notes/troubleshooting/` | 问题排查、报错分析、解决过程 |
| `notes/projects/` | 具体项目相关的阶段记录、方案、任务说明 |
| `templates/` | Markdown 笔记模板 |
| `assets/` | 图片、截图、附件、日志等辅助资料 |

如果后续笔记数量增加，可以再根据实际情况细化目录；如果某类内容较少，也可以暂时不创建对应子目录。

## 5. 文件命名建议

为便于排序、搜索和长期维护，建议 Markdown 文件采用统一命名方式：

```text
YYYY-MM-DD_主题关键词.md
```

例如：

```text
2026-05-20_Git本地工程同步到GitHub.md
2026-05-20_Docker常用命令备忘.md
2026-05-20_MaixCAM串口调试记录.md
2026-05-20_RK3588前端环境配置.md
```

对于项目类笔记，也可以在文件名前增加项目标识：

```text
SMT_2026-05-20_第七阶段开发计划.md
FPV_2026-05-20_Teensy与飞控串口验证.md
RV1126_2026-05-20_RTSP推流问题排查.md
```

推荐原则：

- 文件名尽量清晰，不使用过于笼统的名称；
- 同一类笔记尽量保持命名风格一致；
- 使用中文命名可以提高可读性；
- 重要笔记可以在文件名中加入项目名或技术关键词；
- 避免使用空格和特殊符号，减少跨平台同步问题。

## 6. 单篇笔记推荐格式

每篇 Markdown 笔记建议采用相对固定的结构，便于后续阅读和检索。

```markdown
# 笔记标题

## 1. 背景

记录这篇笔记产生的场景、项目背景或问题来源。

## 2. 核心结论

用简短文字说明最终结论，方便以后快速查看。

## 3. 操作步骤 / 处理过程

记录具体操作、命令、配置、代码或排查过程。

## 4. 问题原因

如果是问题排查类笔记，说明问题出现的原因。

## 5. 解决方法

记录最终有效的解决方案。

## 6. 注意事项

记录后续容易踩坑的地方。

## 7. 参考资料

记录相关链接、文档、命令来源或其他补充资料。
```

对于简单备忘，不必强制使用完整结构；对于重要技术问题、项目方案和开发流程，建议尽量写完整。

## 7. Git 与 GitHub 同步方式

本工程建议使用 Git 进行版本管理，并同步到 GitHub 进行备份。

常用流程如下：

```bash
# 查看当前文件变更状态
git status

# 添加全部变更
git add .

# 提交变更
git commit -m "Update notes"

# 推送到 GitHub
git push
```

如果是首次将本地目录同步到 GitHub，可参考如下流程：

```bash
# 进入 KnowledgeVault 目录
cd KnowledgeVault

# 初始化 Git 仓库
git init

# 添加远程仓库地址
git remote add origin <你的GitHub仓库地址>

# 添加文件
git add .

# 首次提交
git commit -m "Initial commit"

# 设置主分支名称
git branch -M main

# 推送到 GitHub
git push -u origin main
```

后续日常维护时，只需要定期执行：

```bash
git add .
git commit -m "Update knowledge notes"
git push
```

## 8. 使用原则

为了让本工程长期可维护，建议遵循以下原则：

1. **先记录，再整理**  
   开发或学习过程中，先把关键信息记录下来，不必一开始追求完美结构。

2. **重要结论前置**  
   每篇笔记尽量在前面写清楚核心结论，避免以后重新阅读大量过程内容。

3. **保留有效命令和报错信息**  
   对于开发调试类笔记，应尽量保留真实命令、日志、报错和解决步骤。

4. **避免只写现象，不写结论**  
   笔记不仅要记录“发生了什么”，还应记录“为什么发生”和“最后怎么解决”。

5. **定期提交到 GitHub**  
   避免只保存在本地，建议每次整理完一批笔记后及时提交和推送。

6. **不要上传敏感信息**  
   不要在公开仓库中保存密码、密钥、Token、服务器地址、合同隐私信息、个人身份证件信息等敏感内容。

## 9. 敏感信息注意事项

如果本仓库同步到 GitHub，尤其是公开仓库，需要特别注意以下内容不应提交：

- API Key、Token、密码；
- SSH 私钥；
- 服务器公网 IP、账号和口令；
- 单位内部文件、涉密资料、合同敏感条款；
- 未公开项目的核心技术细节；
- 个人隐私信息；
- 第三方不允许公开传播的资料。

如需保存敏感内容，建议使用单独的私有仓库、本地加密目录，或仅在本地保存。

可在 `.gitignore` 中忽略部分本地文件，例如：

```gitignore
# Local private files
.private/
secrets/
*.key
*.pem
*.env

# System files
.DS_Store
Thumbs.db

# Temporary files
*.tmp
*.bak
```

## 10. 后续规划

当前 `KnowledgeVault` 主要作为 Markdown 笔记仓库和 GitHub 备份工程使用。后续如果时间允许，可以逐步扩展为本地智能知识库系统。

可能的后续方向包括：

### 10.1 笔记索引与全文搜索

为本地 Markdown 笔记建立索引，支持基于关键词的快速搜索。

可能技术：

- SQLite
- Whoosh
- ripgrep
- Elasticsearch / OpenSearch

### 10.2 本地大模型问答

将 Markdown 笔记接入本地大模型，实现自然语言问答。例如：

- “帮我找一下之前 Docker 安装清华源的记录”
- “总结一下我之前排查 MaixCAM 串口问题的过程”
- “根据历史笔记，整理一份 RK3588 环境配置流程”
- “我之前关于 GitHub 同步本地工程的步骤是什么？”

可能技术：

- Ollama
- Qwen / DeepSeek / Llama 系列本地模型
- FastAPI
- React / Vue 前端

### 10.3 RAG 检索增强生成

通过向量数据库对 Markdown 笔记进行语义检索，再结合大模型生成回答。

可能技术：

- LlamaIndex / LangChain
- Chroma / Qdrant / FAISS
- bge-m3 / nomic-embed-text / Qwen Embedding
- Ollama API

### 10.4 可视化前端

开发一个轻量化本地前端，用于查看、搜索和问答。

初步功能设想：

- 笔记列表；
- Markdown 预览；
- 标签筛选；
- 项目筛选；
- 全文搜索；
- AI 问答；
- 回答来源引用；
- 最近更新记录。

### 10.5 自动摘要与标签

未来可以使用大模型对新增笔记自动生成摘要、关键词和标签，减少人工整理成本。

## 11. 当前阶段定位

当前阶段，本工程的定位是：

> 一个以 Markdown 为基础、以 Git/GitHub 为备份手段、面向开发学习过程记录的个人知识库仓库。

它不追求复杂功能，重点是保证：

- 内容可持续积累；
- 文件格式长期可读；
- 本地和云端均可备份；
- 后续具备接入大模型和智能检索系统的基础。
